Agent Architecture

A framework for building reliable LLM agents: from foundation theory through puzzle pieces, context management, routing modes, and the feedback loop that makes it agentic.

The Map is Not the Territory

An LLM trains on human writing: descriptions of the world rather than the world itself. Generation produces text consistent with patterns absorbed during training. Whether those patterns correspond to what is actually true is a question the output cannot directly verify.

Hover objects or tokens to trace the mappings between reality and language
Analogy
In Good Will Hunting, Will has consumed an extraordinary breadth of knowledge from books (history, math, philosophy) and can recall and articulate it with total fluency. But when Sean Maguire challenges him, it's on the gap between reading about something and having lived it. Will can quote every line about love, war, loss, but Sean points out he's never held someone dying in his arms, never sat with a sick friend in a hospital. The model is Will before the breakthrough: encyclopedic recall, zero experience. The confidence of the output has nothing to do with whether it corresponds to reality.
Where Reliability Lives
Reliability concentrates in the domains where description and reality converge: code that runs, database queries that execute, APIs that return deterministic results. Outside those domains, grounding tends to benefit from a human in the loop.

Three Ways It Goes Wrong

Most failures fall into three structural types, each benefiting from a different fix. Treating everything as "hallucination" misses the underlying cause, and choosing the wrong mitigation can make the problem worse.

The Three Failure Modes
1. Attention Blindspot: The info was there, but the model looked past it
2. Self-Contradiction: Locally coherent, globally incoherent
3. Knowledge Gap: Confident answers from limited knowledge

Each arises from a different structural property of how language models work. Each typically benefits from a different architectural response.
Context Bloat Amplifies Everything
When the context window is packed with too many tokens (long system prompts, verbose tool outputs, full conversation history) the model's attention is spread thin across everything. The key fact doesn't just compete with its neighbors; it competes with thousands of tokens that have nothing to do with the current question. More context isn't always better context.

1. Attention Blindspot

The info was there, but the model looked past it

The relevant information is in the context window. Attention distributes unevenly across position: heavier at the edges, thinner in the middle. A fact buried at position 40,000 carries a fraction of the weight it would at position 100.

Drag the highlighted fact to different positions to see how attention changes
Example
Someone calls support and spends five minutes on their life story (childhood, career changes, three hobbies) before finally mentioning the one thing they actually need help with. By the time the point arrives, you've stopped listening closely. The model has the same problem: everything before the key fact diluted the attention budget, and the actual question lands where focus is thinnest.

2. Self-Contradiction

Locally coherent, globally incoherent

Each section of a response can be internally sound. The model draws on the right patterns, applies the right logic, and produces text that reads well in isolation. The problem surfaces at the seams: where two sections share an assumption, they may assign it different values. Each patch of the response is locally coherent; the response as a whole is not.

Why this matters
The model generates each section by drawing on the strongest local patterns for that topic. The tax section follows Single-filing patterns. The FSA section follows family-plan patterns. Each is well-formed on its own. But the two sections share an underlying assumption (the employee's family situation) and assign it different values. The contradiction lives at the seam, where the sections overlap on a shared fact. Structured workflow pieces address this by carrying shared context through each step, so assumptions established early stay consistent throughout.

3. Knowledge Gap

Confident answers from limited knowledge

The model is working territory with thin training coverage: a niche domain, an internal process, a company-specific term. The model extrapolates from the nearest patterns it knows; the output arrives confident, and the foundation underneath is slim.

Hover over the sparse side to see the model's confidence gap
Example
You ask the agent about your company's off-cycle bonus payment rules. The model has never seen your company's specific off-cycle configuration, but it has seen thousands of payroll configurations. It produces a plausible answer with the right structure and tone, drawn from those patterns — rather than using search_payroll_calculations and get_pay_off_cycle_types to retrieve the actual data. The confidence is inherited from similar-looking domains, not from actual knowledge of your payroll setup.

Workflows map specific tool call sequences so the model follows a known path. Tools execute deterministically at the leaves. Skills provide domain reasoning when the workflow delegates a decision to the LLM. The rest of this tab covers how these pieces are structured, composed, and maintained.

The Core Building Block

The core unit of composition is the workflow, expressed as a mermaid diagram in a markdown file. The diagram is simultaneously documentation for humans and executable instructions for the model. One artifact, two audiences, zero drift.

The Abstraction Spectrum

Not all workflows need the same level of control. At the deterministic end, every node is a specific tool call, and the LLM follows the flowchart exactly. At the compositional middle, nodes reference other workflows, building larger sequences from verified sub-sequences. At the LLM-bridged end, the model picks the path, choosing which branch to follow or which sub-workflow to invoke based on context.

This spectrum is also where skills find their place. When a workflow is fully deterministic, skills are unnecessary because the diagram already specifies every step. When the workflow delegates decisions to the LLM, skills provide the domain reasoning and interpretation patterns the model draws on to make those decisions well. They live at the higher-abstraction end of the spectrum: the reasoning layer that makes LLM-bridged workflows reliable.

Abstraction spectrum

Two kinds of pieces live in the library
Forward pieces (workflows) map the intended path: the steps that move work toward a result.
Recovery pieces (anti-workflows) encode what to do when a tool returns something unexpected. Covered in the next tab.

Composition

Pieces exist at every granularity. A node in a high-level intent flow expands into a full workflow at the scale below, and each of that workflow's nodes invokes an MCP tool call at the bottom. The same diagram format applies at every level.

Each piece carries a compact identifier (emoji or structured markers) small enough to scan across the entire library without loading the full file. Words like "research", "investigate", and "look up" cluster near the same coordinates in embedding space, but emoji and structured markers occupy regions that natural language leaves sparse. This keeps pieces with similar descriptions separable at retrieval. When a match is found, the piece loads into context.

When a Piece Stops Fitting

Territory changes. A policy update, a new API version, a process that evolved. The piece that worked last month now misfires, and everything built on top of it may need re-examination.

The Ralph Loop

A typical agent loop is straightforward: load a workflow, call tools, collect results. That works until a tool returns something unexpected. Without a recovery path, the agent either retries blindly or gives up. The Ralph loop adds a second channel. When the forward loop hits a problem, a recovery piece (anti-workflow) reads the tool's response through domain-specific skills and returns the agent to a state where the forward loop can resume. The system heals itself rather than stalling.

Why "anti-workflow"?
A workflow moves work forward through a known path. An anti-workflow picks up when that path needs course correction: the tool returned something unexpected, a constraint surfaced, a partial result needs interpretation. It is still a prompt file, but its purpose is recovery rather than forward progress. It draws on domain skills to interpret what the tool returned and decide what to do next. The tool search_workers_by_name_or_id returns "no workers found for that name." The anti-workflow says "try search_workers with partial match, or ask the user to verify the employee ID, because WIDs must be retrieved before calling get_payroll_results."

Two Kinds of Pieces

The atlas holds two primary kinds of puzzle pieces. Forward pieces (workflows) encode the intended work: the happy-path processes, the steps that move things forward. Recovery pieces (anti-workflows) address everything else: error responses, constraints, the failure types described earlier. Both live in the same atlas, retrieved the same way.

When a workflow runs and calls a tool, the tool returns a result. Success, a validation shape, a constraint, partial data, or an error. Recovery pieces exist to handle the full range of those responses, not just the clean path.

Response Pieces Live in the Same Atlas

In this architecture, there is no separate system to handle what tools return. Validation shapes, partial completions, constraints that trigger a reroute, errors that need a retry: all are pieces in the atlas, stored and retrieved alongside forward pieces.

These pieces can call other pieces. Validating an input, retrying a step, adapting a shape, escalating to a human: each is a piece that composes rather than logic that gets improvised on the spot.

Like
A new employee's onboarding covers not just their core responsibilities but also the full range of situations they'll encounter: what to do when a record is incomplete, when a process stalls mid-way, when a rule blocks the usual path. The same handbook, different sections.

How Pieces Work with Tool Responses

A piece reads what the tool returned and acts on it. The response content is the input: specific validation fields, a partial count, a constraint reason.

Retry Limits

Each loop adds to the context window. A retry limit, set in the metaprompt or in code, caps the cycles before escalating to a human.

Unrecognized Responses Go to Mode C

When a response shape has no matching piece in the atlas, the agent halts rather than improvises. It drafts what a piece for this situation might look like.

Tools return messages. Pieces act on them. The atlas grows the same way the rest of the system does: one verified piece at a time.

Built Dynamically, Only as Needed

The context window is assembled fresh for each task. What goes in determines what comes out; what stays out carries little direct weight for this generation, regardless of what the model absorbed during training.

Every generation works from the contents of the context window. Memory, skills, workflows, documents, and tool results are all strategies for assembling the right workspace. Think of it like trying to focus on one task at a cluttered desk: a browser tab left open from something unrelated catches your eye mid-sentence (that's context rot, content competing for attention without earning it), while a desk covered in last week's papers buries the one note you actually need (that's context bloat).

Context Rot pulls attention sideways. Something loosely related, added in good faith, steers generation toward things adjacent to the task rather than central to it. It doesn't need to be wrong to cause drift; it just needs to be there.

Context Bloat buries important content out of reach. Attention concentrates at the edges of a long window and thins in the middle. A critical fact at position 40,000 carries a fraction of the weight it would at position 100.
This is why context is assembled dynamically: the relevant pieces (skills, workflows, or both), user preferences, retrieved documents, and tool results are chosen for this task rather than carried over from previous ones. What the model sees is curated, not accumulated.

Subagents Keep the Main Window Clean

Subagents apply in two scenarios. When a piece has independent branches that can run in parallel, each branch gets its own subagent. When a query requires composing multiple pieces into a single answer (the pattern shown in the Atlas section's "Composing Pieces for a Query"), each piece runs in its own subagent as well. In both cases, the subagent runs in its own context window, carrying only the piece file and its required inputs. It does its work and returns a conclusion, not a transcript. The main agent's window accumulates findings rather than the intermediate state each piece generates.

Compaction: Summarise as You Go

Over a long session, the window accumulates: tool results, intermediate outputs, recovery traces. Compaction periodically collapses accumulated history into a short digest, archiving the full trace to memory. The window resets to something the agent can see clearly.

One Agent, Four Modes

The agent operates primarily as a retrieval engine: find the right piece, inject it, run it. Mode A is the common case. The middle two modes, Planning and Cartography, handle assembly and new territory. Mode D closes the loop when the query itself needs narrowing first.

A

The Librarian

One piece fits cleanly

A single piece matches the query with high confidence. The agent injects it and executes. The output is shaped by the piece rather than by open-ended generation.

The Clarifier

Query needs narrowing first

The query is broad enough that routing to any piece would be a guess. The agent surfaces what it would need to route confidently, then re-routes once the space has narrowed.

Like

B
Agent-discretion assembly

The Orchestrator

Multiple pieces or parallel branches, subagent per unit

The query spans several existing pieces or branches. The planner identifies which pieces apply and whether any branches can run in parallel, then spawns a subagent for each one. Each subagent runs its piece in an isolated context window, carrying only what that piece requires. It returns a conclusion, not a transcript.

The risk, and its ceiling
Each subagent runs verified pieces; the risk concentrates at the merge. For sequential compositions, the interface between pieces may not match. For parallel ones, individually correct findings can conflict when joined.
C
Drafts before unknown territory

The Cartographer

Uncharted territory

When no piece in the atlas matches with enough confidence, the agent halts and drafts rather than extrapolates: scope, likely structure, connections to adjacent pieces. Whether that draft proceeds, queues for review, or waits on a human depends on configuration.

B and C: where the unverified surfaces
In both modes, the agent reaches a point where proceeding means generating something unverified: a joint in B, an entire piece in C. What happens next depends on the domain, the stakes, and how the system is configured.
Modes B, C, and D all involve the agent reaching the edge of what's verified. The form varies: an uncertain joint, unknown territory, or an ambiguous query. D is last because it resolves by routing into A, B, or C.

How Pieces Relate in a Composition

In sequential compositions, fragility concentrates at the interface: a schema change, a renamed field, a shape the agent gets wrong. In parallel compositions, pieces work on their own inputs and the agent joins results into a response.

What the pieces share is that each is verified within its own scope. The unverified part is whatever the agent generates to connect them: an explicit interface mapping, a merge of independent results, or a judgment call about which findings to surface.

Gaps Are Meaningful

A gap marks territory the action space hasn't covered yet: no success path, no recovery path, no verified piece. That boundary is itself useful; it shows where coverage ends and where the next piece belongs.

Two Kinds of Signal

Memory loads into the context window at session start. What distinguishes it is where it comes from: not a static store, but a distillation of prior sessions, shaped by what worked, what broke, and how the user prefers to work.

Execution History
What went well, and where things broke

Successful sequences are logged as proven routes, with enough context to recognize similar situations later. Failures (wrong pieces, broken joints, coverage gaps) are logged alongside them.

Failures are archived rather than deleted. A removed failure record means the system will likely encounter the same situation again with no warning.

Written at session end · both success and failure

User Profile
How this person prefers to work

The profile captures two things: explicit preferences stated by the user, and patterns observed across sessions. Output style covers length, format, and vocabulary. Interaction style covers pacing, confirmation behavior, and how much clarification to seek upfront.

The profile loads at session start, so the agent can calibrate without asking the same questions twice.

Written on correction, override, or observed pattern

The Review Cycle

Pieces go stale. A policy changes, a process shifts, a skill or workflow that worked reliably starts producing the wrong results. Memory entries age. User preferences drift. The question after any outcome is the same: was this a one-off, or does something need updating?

This cycle is what drives atlas evolution. The Day 1, Day 30, and Day 100 snapshots in the Atlas tab are the cumulative result of these review passes: pieces added, recovery paths filed, stale pieces retired, gaps filled.

Seven Design Principles

The architecture holds because of a small set of commitments. Each is a consequence of understanding what an LLM actually is, and where its reliability comes from.

The first six address the three structural properties of an LLM. The seventh is what makes it agentic.

Context: Relevance and Signal

1

Fill for Relevance, Not Completeness

Memory, skills, workflows, retrieved documents, tool results: each is a choice about what belongs on the workbench for this task. The context window is the primary workspace the model works from. Most items either earn their place, pulling generation toward the task, or compete for attention that should go elsewhere.

2

Keep the Signal Visible as the Session Grows

A well-filled window at the start can degrade as a session runs. Several mechanisms defend it: subagent scoping, compaction, retry limits, and compact piece identifiers. Each keeps the window from filling with intermediate state that crowds out the content attention should reach.

Interfaces: Joints and Judgment

3

The Generated Parts Carry the Risk

Pieces carry their own guarantees; what the agent generates to connect them does not. Whether the composition is sequential or parallel, the generated layer is the right place to focus review and testing.

4

Hard Limits Live at the Tool Layer

In this design, what the agent can and cannot do is enforced in software at the tool level rather than by routing logic asking permission. Human judgment is available at the moments it genuinely matters: novel joints that recovery loops can't resolve, frontier territory with no draft path, decisions where the output's purpose is outside what the system can verify.

Coverage: Structure and Atlas

5

Retrieval Before Improvisation

Existing, tested pieces carry their own reliability. The less the agent generates from scratch, the more its output inherits that reliability. Where the atlas has a gap, that visibility is itself useful; it marks where coverage should grow next.

6

Prompt Files Are Pieces; Tools Are Code

The core building block is a workflow expressed as a mermaid diagram. At the deterministic end, every node is a tool call and the path is fixed. At the LLM-bridged end, the model picks the path, guided by skills that provide domain reasoning. Tools are the leaf operations that workflow nodes invoke. Recovery pieces follow the same structure, composing retry, adapt, and escalation logic. All pieces live in the same atlas.

Agency: The Feedback Loop

7

Every Run Teaches the Atlas

A static system retrieves. An agentic one updates. Without a feedback loop, the atlas tends to cover only what was anticipated at setup. With one, each session can add to it: successful routes become proven paths, handled failures become recovery pieces, Mode C drafts get reviewed and filed as new pieces. The coverage of the system on day 100 looks different from its coverage on day one. That shift, from static retrieval to continuous learning, is a key characteristic of what makes a system agentic.

Three structural failure modes follow from what a language model is. This architecture routes around each. It fills the window with relevant, well-positioned content so attention reaches what matters, enforces hard limits at the tool layer, and grounds execution in retrieved, verified pieces wherever the atlas has coverage.
